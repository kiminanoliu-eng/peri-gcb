# 经验教训总结

## 文档说明

本文档总结从每个错误中学到的教训、改进的工作方法和成功案例的最佳实践。

**目的**: 将错误转化为知识，持续改进工作流程

---

## 核心教训

### 教训 1: 质量永远优先于速度

**背景**:
- 为了提高速度，曾同时处理5-7个产品
- 结果产生大量错误数据，需要返工修复
- 实际上反而降低了整体效率

**学到的**:
- ✅ 一次只做1个产品
- ✅ 每个产品完成后让用户验证
- ✅ 慢即是快：做对一次比返工多次更快

**应用**:
```
错误方式: 同时处理5个产品 → 5个都有错误 → 返工5次 = 10个产品的工作量
正确方式: 逐个处理5个产品 → 每个都正确 = 5个产品的工作量
```

**数据支持**:
- 批量处理: 5个产品，3个有错误，返工率60%
- 逐个处理: 最近2个产品（LIWA, SB），0个错误，返工率0%

---

### 教训 2: 验证正确性，不只是有效性

**背景**:
- UNO产品的PDF链接返回HTTP 200，可以打开
- 但PDF内容不是UNO产品的手册
- 只验证了"链接有效"，没有验证"内容正确"

**学到的**:
- ✅ HTTP 200 ≠ 内容正确
- ✅ 必须打开PDF确认内容
- ✅ 必须确认项目在产品页面上
- ✅ 必须确认视频是该产品的介绍

**应用**:
```
有效性验证: curl -I {url} → HTTP 200 ✓
正确性验证: 打开文件/页面 → 内容是该产品的 ✓
```

**三步验证法**（以PDF为例）:
1. 验证链接有效（HTTP 200）
2. 验证文件类型（Content-Type: application/pdf）
3. **验证内容正确**（打开PDF确认是该产品的手册）

---

### 教训 3: 不要依赖工具的不完整输出

**背景**:
- WebFetch返回"未找到项目"
- 没有验证就相信了
- 实际上用curl可以找到项目

**学到的**:
- ✅ WebFetch可能返回不完整的HTML
- ✅ 必须用curl获取完整HTML源代码
- ✅ 不要100%信任工具输出，要验证

**应用**:
```bash
# 错误方式
WebFetch → "未找到项目" → 相信 → 标记为0个项目

# 正确方式
WebFetch → "未找到项目" → 怀疑 → curl验证 → 找到项目
```

**原则**: 工具是辅助，验证是必须

---

### 教训 4: 严格遵守命名规则

**背景**:
- 创建了`handset_alpha_complete.json`
- 但slug是`handset-alpha`
- 代码找不到文件

**学到的**:
- ✅ 文件名必须与slug完全匹配
- ✅ 连字符`-`不能替换成下划线`_`
- ✅ 不能使用简短版本
- ✅ 创建文件前必须确认slug

**应用**:
```bash
# 创建文件前的标准流程
1. 从products_v2.json确认slug
2. 记录完整slug（包括所有连字符）
3. 构建文件名: ${slug}_complete.json
4. 验证文件名格式
5. 创建文件
```

**记忆技巧**: "slug是什么，文件名就是什么_complete.json"

---

### 教训 5: 只做用户明确要求的工作

**背景**:
- 用户说"pdf链接不对"（指UNO）
- 误解为所有产品都有问题
- 擅自修改了HANDSET和VARIO
- 用户强烈反对

**学到的**:
- ✅ 先询问用户具体哪个产品有问题
- ✅ 只修改用户明确指出的产品
- ✅ 不要假设问题的范围
- ✅ 已验证的产品 = 正确的产品

**应用**:
```
用户: "pdf链接不对"
错误反应: 假设所有产品都有问题 → 擅自修改多个产品
正确反应: "请问是哪个产品的PDF链接不对？" → 只修改该产品
```

**原则**: 
- 不擅自修改已完成的产品
- 不擅自修改代码
- 有疑问立即询问用户

---

### 教训 6: 数据来源必须准确

**背景**:
- MAXIMO使用了"港珠澳大桥"项目
- 但该项目不在MAXIMO的产品页面上
- 项目是从china_projects.json随机选择的

**学到的**:
- ✅ 项目必须从产品页面HTML提取
- ✅ 不从china_projects.json随机选择
- ✅ 不使用其他产品的项目
- ✅ 数据来源必须可追溯

**应用**:
```bash
# 正确的项目提取流程
1. curl获取产品页面HTML
2. grep提取项目链接
3. 验证项目确实在HTML中
4. 逐个提取项目详细信息
```

**原则**: "数据从哪里来"比"数据是什么"更重要

---

### 教训 7: 在正确的地方搜索

**背景**:
- 在YouTube主页搜索产品名称
- 找到的大多是德语培训视频
- 不符合要求

**学到的**:
- ✅ 必须在PERI官方频道内搜索
- ✅ 不在YouTube主页搜索
- ✅ 搜索位置影响搜索结果质量

**应用**:
```
错误: YouTube主页搜索 → 所有上传者的视频 → 大量不相关结果
正确: PERI频道内搜索 → 只有PERI的视频 → 相关结果
```

**原则**: 在正确的范围内搜索，而不是在最大的范围内搜索

---

## 改进的工作方法

### 方法 1: 分阶段验证

**旧方法**:
```
收集所有数据 → 创建JSON → 生成HTML → 部署 → 发现错误
```

**新方法**:
```
收集数据 → 验证数据 → 创建JSON → 验证JSON → 生成HTML → 验证HTML → 部署 → 在线验证
```

**改进点**:
- 每个阶段完成后立即验证
- 早发现错误，早修正
- 避免错误累积

---

### 方法 2: 三步验证法

**应用场景**: PDF链接、项目案例、YouTube视频

**步骤**:
1. **有效性验证**: 链接/资源是否存在
2. **类型验证**: 是否是正确的类型（PDF/项目/视频）
3. **内容验证**: 内容是否是该产品的

**示例（PDF）**:
```bash
# 步骤1: 有效性
curl -I {pdf_url} → HTTP 200 ✓

# 步骤2: 类型
Content-Type: application/pdf ✓

# 步骤3: 内容
打开PDF → 确认是该产品的手册 ✓
```

---

### 方法 3: 使用检查清单

**旧方法**:
- 凭记忆处理每个产品
- 容易遗漏步骤
- 不同产品处理方式不一致

**新方法**:
- 使用PREVENTION_CHECKLIST.md
- 每个产品都执行相同的检查
- 确保一致性和完整性

**效果**:
- LIWA和SB产品使用检查清单后，0错误
- 之前的产品没有使用检查清单，多个错误

---

### 方法 4: 自动化验证

**工具**: verify_product.py

**功能**:
- 自动验证文件名
- 自动验证JSON结构
- 自动验证URL有效性
- 自动验证HTML生成

**使用时机**:
- JSON文件创建后
- HTML生成后
- 部署前

**效果**:
- 减少人工验证工作量
- 提高验证准确性
- 及早发现错误

## 工作流程的演进

### 第1版（初期）
```
选择产品 → 收集数据 → 创建JSON → 生成HTML → 部署
```
**问题**: 
- 批量处理多个产品
- 验证不充分
- 错误率高

---

### 第2版（改进后）
```
用户指定产品 → 确认slug → 收集数据 → 验证数据 → 创建JSON → 生成HTML → 部署 → 用户验证
```
**改进**:
- 一次只做1个产品
- 增加数据验证
- 用户最终验证

---

### 第3版（当前）
```
用户指定产品
  ↓
确认slug（从products_v2.json）
  ↓
收集数据（curl，不用WebFetch）
  ↓
三步验证法（有效性 + 类型 + 内容）
  ↓
创建JSON（严格命名规则）
  ↓
验证JSON（verify_product.py）
  ↓
生成HTML（rebuild_site_v2.py）
  ↓
验证HTML（verify_product.py）
  ↓
部署（deploy-peri-gcb.command）
  ↓
在线验证
  ↓
用户最终验证
```
**改进**:
- 使用检查清单
- 自动化验证
- 多层验证
- 错误率接近0
