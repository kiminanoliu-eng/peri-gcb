# 产品网站生成项目规则

## 项目背景
为PERI GCB创建160个产品的中文介绍网站，每个产品页面需要包含：
1. 7种语言翻译（中、英、西、德、葡、塞、匈）
2. PDF下载链接
3. YouTube视频（英语、10分钟内、最新）
4. 产品相关的项目案例
5. Web3Forms留言板

## 核心原则

### ⚠️ 质量优先于速度
**错误案例**: 在批量处理5-7个产品时，为了追求速度，依赖工具输出而不验证，导致：
- MAXIMO产品只提取了1个项目，实际有4个
- 提取的项目（港珠澳大桥）根本不在MAXIMO页面上
- 其他产品直接标记为"无项目案例"，没有仔细检查

**正确做法**: 
- 一次处理1个产品，完成后让用户验证
- 每个产品都要仔细检查，不能因为批量处理而降低质量

## 项目案例提取规则

### 规则1: 必须验证项目确实属于该产品
**错误**: 使用了之前WebFetch缓存的结果（港珠澳大桥），没有验证是否在当前产品页面上

**正确流程**:
```bash
# 1. 直接获取HTML源代码
curl -s "https://cn.peri.com/products/[产品slug].html" > temp.html

# 2. 提取所有项目链接
grep -o 'href="https://cn.peri.com/projects/[^"]*"' temp.html

# 3. 验证项目数量
grep -c "project-teasers__item" temp.html

# 4. 逐个提取项目信息
grep -A 20 "project-teasers__item-regular-list-headline" temp.html
```

### 规则2: 不能依赖WebFetch的部分输出
**问题**: WebFetch工具可能只返回页面的部分内容，特别是动态加载的项目卡片

**解决方案**:
- 使用curl + grep直接解析HTML
- 提取所有`<li class="project-teasers__item">`元素
- 对每个项目提取：
  - 项目名称 (`project-teasers__item-regular-list-headline`)
  - 位置 (`project-teasers__item-regular-list-kicker`)
  - 描述 (`project-teasers__item-regular-list-description`)
  - 图片URL (`<img src="..."`)
  - 链接 (`href="https://cn.peri.com/projects/..."`)

### 规则3: 区分"无项目"和"未检查"
**错误**: 当WebFetch返回"no projects found"时，直接标记为无项目案例

**正确做法**:
- 如果WebFetch说"无项目"，必须用curl验证
- 检查HTML中是否有`<h2>项目实例</h2>`标签
- 检查是否有`project-teasers__items-list`类
- 只有确认HTML中确实没有项目卡片，才能标记为"无项目"

## YouTube视频搜索规则

### 规则4: 搜索关键词必须包含"formwork"
**错误**: 只搜索"peri [产品名]"，结果大多是德语培训视频

**正确做法**:
```bash
# 搜索时加上formwork关键词
curl -s "https://www.youtube.com/results?search_query=peri+[产品名]+formwork"
```

### 规则5: 验证视频时长和语言
**要求**:
- 视频时长 ≤ 10分钟（600秒）
- 优先选择英语视频
- 优先选择产品介绍视频，而非培训视频
- 检查视频发布时间，选择最新的

**验证方法**:
```bash
# 提取视频时长
curl -s "https://www.youtube.com/watch?v=[VIDEO_ID]" | grep -o '"approxDurationMs":"[^"]*"'

# 提取视频标题（判断语言）
curl -s "https://www.youtube.com/watch?v=[VIDEO_ID]" | grep -o '"title":{"runs":\[{"text":"[^"]*"'
```

## PDF链接搜索规则

### 规则6: 扩大搜索范围到所有PERI区域网站
**错误**: 只在peri.id（印尼）网站搜索，导致GRIDFLEX找不到PDF

**正确做法**:
搜索多个区域网站：
- peri.id (印尼)
- peri.com.au (澳大利亚)
- peri.ltd.uk (英国)
- peri.com (全球)
- peri.de (德国)

**搜索方法**:
```bash
# 方法1: 使用WebSearch
site:peri.id OR site:peri.com.au OR site:peri.ltd.uk "[产品名]" pdf

# 方法2: 直接访问产品页面查找
curl -s "https://www.peri.id/products/[产品slug].html" | grep -o 'href="[^"]*\.pdf"'
```

### 规则7: 验证PDF链接有效性
**要求**:
- PDF链接必须以`.pdf`结尾
- 不能是Scribd或其他第三方网站的链接
- 必须是PERI官方网站的直接下载链接

## 工作流程规则

### 规则8: 每个产品完成后必须验证
**错误**: 批量生成5-7个产品后才让用户检查，导致错误累积

**正确流程**:
1. 完成1个产品
2. 生成HTML页面
3. 部署到GitHub Pages
4. 让用户验证
5. 用户确认无误后，继续下一个产品

### 规则9: 创建验证清单
每个产品完成后，必须检查：
- [ ] PDF链接：是否存在？是否可访问？是否是该产品的PDF？
- [ ] YouTube视频：是否存在？时长是否≤10分钟？是否英语？是否该产品的视频？
- [ ] 项目案例：数量是否正确？是否确实在cn.peri.com该产品页面上？
- [ ] 7种语言翻译：是否完整？
- [ ] Web3Forms：是否配置正确？

### 规则10: 记录每个产品的状态
创建一个tracking文件，记录：
```markdown
| 产品名称 | PDF | YouTube | 项目数 | 状态 | 备注 |
|---------|-----|---------|--------|------|------|
| MAXIMO  | ✅  | ✅      | 4      | ❌   | 需要重做，项目提取错误 |
```

## 数据验证规则

### 规则11: 交叉验证数据来源
**错误**: 使用了缓存或之前的WebFetch结果，没有验证数据来源

**正确做法**:
- 每次提取数据时，记录数据来源URL
- 在JSON文件中添加`data_source`字段
- 生成页面前，验证数据来源与当前产品匹配

### 规则12: 不要假设页面结构一致
**错误**: 假设所有产品页面结构相同，使用相同的提取方法

**正确做法**:
- 每个产品页面可能有不同的HTML结构
- 先检查页面结构，再决定提取方法
- 如果提取失败，尝试不同的CSS选择器或XPath

## 错误处理规则

### 规则13: 遇到问题立即停止，不要继续
**错误**: 发现MAXIMO提取有问题，但继续处理其他产品

**正确做法**:
- 发现任何数据不一致时，立即停止
- 向用户报告问题
- 等待用户确认后再继续

### 规则14: 保持透明，如实报告
**错误**: 说"cn.peri.com上无项目案例"，实际上是没有仔细检查

**正确做法**:
- 如实报告："WebFetch未找到项目，需要手动验证"
- 不要用"无项目"掩盖"未检查"的事实
- 让用户知道哪些是确认的，哪些是需要验证的

---

## 总结

### 核心教训
1. **质量 > 速度**: 宁可慢一点，也要确保每个产品数据准确
2. **验证 > 假设**: 不要假设工具输出是正确的，必须验证
3. **透明 > 隐藏**: 如实报告问题，不要用模糊的说法掩盖
4. **一次一个 > 批量处理**: 每个产品完成后让用户验证
